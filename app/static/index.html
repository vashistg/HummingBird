▶ Why is cosine similarity O(n) — and how does ANN fix it?

Step 1 — Audio → embedding (128 numbers)

librosa extracts 128 MFCC coefficients, averaged over time. Each song becomes a single point in a 128-dimensional space.

mfcc = librosa.feature.mfcc(y, sr, n_mfcc=128) # shape (128, T) embedding = mean(mfcc, axis=time) # shape (128,) ← one point in ℝ¹²⁸

Step 2 — Cosine similarity

Two audio clips "sound similar" when their embedding vectors point in roughly the same direction. We measure this with the angle between them.

cos_sim(a, b) = dot(a, b) / (|a| × |b|) After L2-normalizing all vectors to unit length: cos_sim(a, b) = dot(a, b) ← just a dot product!

Step 3 — Why brute-force is O(n)

To find the best match, we compute the dot product between the query and every song. With n songs and 128 dimensions, that's n × 128 multiplications.

scores = E @ query # E has shape (n, 128) # This is n dot products — one per song # Scale a 10-song library to 10M songs → 10M × 128 ops

Step 4 — hnswlib HNSW: graph traversal (reference impl)

HNSW builds a layered graph — think of it like a highway system. The top layer connects only a few "hub" nodes far apart; lower layers are denser. A query starts at the top and greedily descends, zooming in on the answer.

Build: connect each vector to M=16 nearest neighbors per layer Query: greedy walk from random entry → descend layers → ef=50 candidates Cost: O(log n) hops per layer — orders of magnitude faster than O(n)

Step 5 — FAISS IVF: k-means partitioning

IVF clusters all embeddings into Voronoi cells with k-means. At query time only nprobe nearest cells are searched — the rest are ignored entirely.

n_lists = sqrt(n) Voronoi cells (k-means clustering) nprobe = n_lists / 4 (only 25% of cells searched) Cost: O(nprobe × n/n_lists) ≈ O(sqrt(n)) for default settings

Step 6 — FAISS HNSW vs hnswlib HNSW

Both implement the same algorithm (Malkov & Yashunin 2018). The difference is in the engine: hnswlib is the standalone reference implementation (~300 LOC C++ core, easy to read). FAISS HNSW adds SIMD-accelerated distance kernels and integrates with FAISS's quantization ecosystem — significantly faster at millions of vectors.

hnswlib: standalone, clean API, online insertions (add items after build) FAISS HNSW: same graph structure + AVX2/AVX-512 SIMD kernels + quantization Both: M=connections per node, ef_construction=build quality, ef=query quality

The catch: approximate

IVF and HNSW can miss the true nearest neighbor. The timing table shows whether each method agreed with brute-force — that agreement rate is called recall. With only 12 songs every method should agree; the differences emerge at 1M+ vectors.

SQLite (songs.db)
Stores song metadata (title, artist, filename) and each embedding as a numpy blob. Only brute-force reads this at query time — the ANN engines use their own index files.
128-dim unit vector
128 MFCC coefficients averaged over time, then L2-normalised. L2-normalisation means cosine similarity = dot product, which is what all five engines compute.
Brute Force O(n)
One dot product per song. Scales linearly — double the library, double the time. The ground-truth baseline everything else is measured against.
hnswlib HNSW O(log n)
Reference implementation by the algorithm's original authors. Builds a layered graph; queries greedily descend from the sparse top layer to the dense bottom layer.
FAISS Flat O(n)
Same O(n) as brute force but uses AVX2/AVX-512 SIMD to compute many dot products per CPU cycle. Often faster than numpy despite identical complexity.
FAISS IVF O(√n)
Clusters all vectors into sqrt(n) Voronoi cells with k-means. At query time only nprobe=25% of cells are searched — the rest are skipped entirely.
FAISS HNSW O(log n)
Same HNSW algorithm as hnswlib but with FAISS's SIMD distance kernels and memory layout. Faster at large scale; same accuracy trade-offs.
Index files (data/indices/)
Written once at ingest time, memory-mapped at query time. Rebuilding indices (e.g. after adding songs) doesn't touch the database, and vice versa.
